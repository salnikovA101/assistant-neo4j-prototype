# Документация проекта Neo4j Voice Assistant

## 1. Обзор системы и назначение

Проект представляет собой голосовой (а также текстовый) ИИ-ассистент для интерактивного взаимодействия с графовой базой знаний Neo4j. Ассистент выступает в роли Q&A слоя (Graph-RAG) для данных БД.

Предметная область базы данных: *food-science, микробиология, умная упаковка (smart packaging)*.

**Ключевой принцип работы:** Ассистент опирается строго на данные из базы, всегда сопровождая ответ провенансом (цитатой `evidence`, файлом-источником и `confidence`).

---

## 2. Архитектура и обработка запроса (Pipeline)

Система построена на асинхронном пайплайне, управляемом классом `ServerPipeline` (FastAPI).

### Схема полного пайплайна

```
Аудио-запрос (WAV от клиента)
    │
    ▼ STT (Speech-to-Text)
    │   Faster Whisper переводит речь в текст.
    │   На выходе: строка с вопросом пользователя.
    │
Текстовый-запрос (в случае отправки текстового запроса минуя STT)
    │
    ▼ LLM Orchestrator
    │   LLM принимает текст вопроса и переформулирует его и отдает агенту GraphQA через вызов инструмента ask_database.
    │   Один вызов инструмента ask_database всегда обязателен!!
    │
    ▼ Tool Call: ask_database (GraphQA Agent)
    │   Обращение к графу (Text-to-Cypher).
    │   На выходе: сырые данные из БД что смог достать агент.
    │
    ▼ Оценка полноты данных (LLM)
    │   Если информации хватает ──► переход к Финальному ответу.
    │   Если нет ──► повторный вызов ask_database с новыми вопросами.
    |
    ▼ LLM (Финальный ответ)
    │   Формирует итоговый ответ на основе контекста из БД.
    │   В конец ответа жестко прикрепляется блок GRAPH_NODES: [...]
    │
    ▼ Сплиттинг (ServerPipeline)
    │   Ответ разделяется:
    │   1. Чистый текст ответа.
    │   2. Технический блок GRAPH_NODES (сохраняется в памяти сервера).
    │
    ▼ TTS (Text-to-Speech)
    │   Чистый текст идет в TTS (стриминг PCM-аудио клиенту).
    │   В HTTP-заголовках клиенту передается флаг Has-Graph: true.
    │
    ▼ Клиент (UI)
        Воспроизводит аудио ответа. Показывает пользователю кнопку "Показать граф".

Пользователь жмет "Показать граф"
    │
    ▼ HTTP GET /graph_data
    │   Сервер обращается к сохраненному блоку GRAPH_NODES.
    │
    ▼ GraphFilterAgent
    │   Перехватывает GRAPH_NODES и оригинальный Cypher, 
    │   фильтрует граф для UI.
    │
    ▼ Клиент (UI)
        Отрисовывает отфильтрованный граф с помощью vis-network.
```

## 3. Инструменты и Агенты (Text-to-Cypher)

Это ядро системы извлечения данных. Реализовано в `server/tools/`.

### 3.1 `GraphQA` — Агент извлечения данных
Отвечает за перевод естественного языка в Cypher и получение сырых данных из Neo4j.

```
Пользовательский запрос (внутри Tool Call)
    │
    ▼ Text-to-Cypher (LLM)
    │   Использует cypher_system.md и cypher_examples.md (9 примеров).
    │   Переводит термины на английский, маппит на строгие метки схемы.
    │   На выходе: сырой Cypher-запрос.
    │
    ▼ Run ID Injection
    │   Аппаратно внедряет (regex) активный run_id во все связи.
    │   Предотвращает чтение данных из других экспериментов.
    │
    ▼ Выполнение запроса (Neo4j)
    │   Использует SafeReadOnlyNeo4jGraph.
    │   Принудительный default_access_mode="READ" (защита от CREATE/DELETE).
    │
    ▼ Retry-цикл (до 3 попыток)
    │   Если запрос вернул синтаксическую ошибку или пустой результат,
    │   генерирует фидбек LLM ("Your query returned empty results...")
    │   и отправляет на повторную генерацию Cypher.
    │
    ▼ Возврат сырых данных
        Сырые объекты Neo4j (узлы, связи, confidence, evidence)
        возвращаются Оркестратору в виде JSON (без предварительного текстового форматирования).
```

### 3.2 `GraphFilterAgent` — Агент точной визуализации
Отвечает за то, что в UI отрисовывается ровно тот подграф, о котором ассистент рассказал пользователю.

```
Оригинальный Cypher (от GraphQA) + GRAPH_NODES (от LLM)
    │
    ▼ GraphFilterAgent (LLM)
    │   Использует graph_filter/system.md.
    │   Опираясь на упомянутые сущности (GRAPH_NODES),
    │   модифицирует оригинальный Cypher.
    │
    ▼ Инъекция IN-фильтров
    │   Ко всем узлам оригинального запроса добавляется строгий фильтр:
    │   ... AND m1.name IN [...] AND m2.name IN [...]
    │   Обрезает ветви первоначального широкого запроса.
    │
    ▼ Выполнение запроса (Neo4j)
    │   Выполнение отфильтрованного запроса с RETURN * 
    │   (возвращает полные объекты Node и Relationship).
    │
    ▼ GraphVizExtractor
    │   Парсинг объектов графа, извлечение всех свойств.
    │   Назначение цветовой схемы (Microbe, Metabolite, EC).
    │
    ▼ Web UI
        Отрисовка отфильтрованного графа с помощью vis-network.
```

## 4. Схема Neo4j графа

Граф построен по методологии онтологии DOLCE.

*   **Узлы (3 типа):**
    *   `Microbe` (бактерии, патогены)
    *   `Metabolite` (химические вещества, газы)
    *   `EnvironmentCondition` (температура, влажность, pH)
*   **Связи (5 типов):** `PRODUCES`, `CONSUMES`, `INHIBITS`, `STIMULATES`, `REQUIRES`.
*   **Свойства связей (Обязательный провенанс):**
    *   `evidence`: дословная цитата из PDF.
    *   `source_file`: имя файла-источника.
    *   `chunk_id`: идентификатор текстового фрагмента.
    *   `confidence`: уверенность LLM при экстракции (0.0 - 1.0).
    *   `run_id`: уникальный идентификатор эксперимента/загрузки.

## 5. LLM-подсистема (`server/llm/`)

*   **`BaseLLMProvider` / `OpenAIProvider`:** Вся работа с LLM унифицирована через OpenAI SDK. Это позволяет использовать любые совместимые API (OpenRouter, Gemini, vLLM).
*   **Tool Call Loop:** В методе `generate_response` реализован полноценный цикл обработки вызовов инструментов (до `max_turns`). Если модель решает вызвать `ask_database`, фреймворк выполняет функцию локально и возвращает результат в контекст как `role: tool`.
*   **`HistoryManager`:** Ограничивает контекст диалога (deque), удаляет неудачные вызовы и преобразует историю в формат сообщений OpenAI.
*   **Промпты:** Главная поведенческая логика находится в `prompts/logic.md`. Формат ответа строго контролируется промптами из `prompts/output_quality.md` и `output_speed.md`.

## 6. Речь: STT и TTS

### 6.1 Speech-to-Text (`server/stt/`)
Используется **Faster Whisper** (`large-v3-turbo`).
*   Поддерживает конвертацию `float32` numpy-массивов (локально) и `WAV bytes` (от веб-клиента).
*   Автоматический VAD-фильтр (Voice Activity Detection) отсекает тишину.
*   Warmup-прогон "пустого" аудио при старте сервера исключает задержку при первом обращении.

### 6.2 Text-to-Speech (`server/tts/`)
Управляется `TTSManager` и поддерживает 3 режима горячего переключения:

1.  **`quality` (FasterQwen3TTS)**
    Высокое качество, voice cloning (требует эталонный WAV). Генерация выполняется в отдельном потоке, передавая чанки в `asyncio.Queue` для потоковой передачи. TTFT ~ 500 мс.

    *Ограничения:*
    *   Требует много VRAM (тяжелая модель).
    *   Хорошо читает на русском, но говорит с акцентом на английском.
    *   Плохо читает длинные числа (может путаться в произношении).

2.  **`speed` (Silero V5)**
    Сверхбыстрый синтез (latency ~100мс), малый размер. Время на весь текст ~ 200мс.

    *Ограничения:*
    *   Модель читает **только кириллицу**. Из-за этого промпт LLM переключается на `output_speed.md`, который заставляет LLM транслитерировать все англоязычные термины (DNA → ди эн эй) и писать числа словами.
    *   Более роботизированный голос, менее естественный по сравнению с `quality`.

3.  **`cloud` (Облачные API)**
    Использование OpenAI-совместимых облачных TTS (например, `gpt-4o-mini-tts`). Самый легковесный вариант для локальной машины.
    
    *Ограничения на примере `gpt-4o-mini-tts`:*
    *   Немного Английский акцент на русском языке.
    *   Немного больше время TTFT по срванению с `quality`.

## 7. Серверная архитектура (`server/`)

*   **FastAPI:** Основной фреймворк (`core/app.py`). Поднимает следующие эндпоинты (ручки):

| Метод | Эндпоинт | Назначение и формат данных |
| :--- | :--- | :--- |
| `POST` | `/process` | **Ручка на весь пайплайн.** Принимает сырые `WAV` байты аудио. Возвращает потоковый ответ (`StreamingResponse`) в формате `audio/pcm` (чанками int16, 24kHz). Метаданные (распознанный текст, ответ LLM, флаг `Has-Graph`) передаются через кастомные HTTP-заголовки (URL-encoded). |
| `POST` | `/stt` | Быстрый перевод речи в текст. Принимает `WAV` байты, возвращает JSON: `{"text": "..."}`. Используется клиентами для мгновенного отображения распознанной речи без ожидания ответа LLM. |
| `POST` | `/process_text` | Текстовый аналог `/process`. Принимает JSON `{"text": "..."}`. Возвращает потоковый `audio/pcm` ответ (генерация аудио из текста ответа) с заголовками метаданных. |
| `POST` | `/process_text_test`| Тестовая текстовая ручка. Принимает JSON `{"text": "..."}`, возвращает JSON `{"answer": "..."}`. **Не** генерирует аудио. Полезна для скриптов, E2E тестов и отладки промптов. |
| `POST` | `/graph_data` | Возвращает данные графа для отрисовки в Web UI. Обращается к пайплайну за подграфом, который сформировал агент фильтрации на основе последнего успешного Cypher-запроса. Возвращает JSON с `nodes` и `edges`. |
| `GET` | `/health` | Проверка жизнеспособности (Readiness probe). Возвращает `{"status": "ready"}` после завершения инициализации и прогрева (warmup) моделей. |
| `GET` | `/ui/` | Раздача статики. Веб-интерфейс ассистента с визуализатором графа на базе `vis-network`. |
*   **Lifespan Events:** Инициализация и прогрев моделей, настройка трейсинга происходят при старте сервера.
*   **Трейсинг (OpenTelemetry + Phoenix):** `utils/tracing.py` автоматически инструментирует вызовы OpenAI и создает кастомные спаны (`span.kind = TOOL / CHAIN`) для контроля времени выполнения и отладки промптов. Интерфейс доступен на порту `6006`.

## 8. Клиентская часть

### 8.1 Web UI (`server/static/`)
*   Использует `AudioContext` для записи с микрофона и **мгновенного потокового воспроизведения** PCM-ответа.
*   **Barge-in:** Нажатие на микрофон прерывает ответ ассистента через отправку disconnect-события (сервер прерывает генерацию) и `AbortController`.
*   **Визуализация графа:** `vis-network` с физикой `ForceAtlas2Based` для плавного разлета узлов.

### 8.2 Desktop Client (`client/`)
Альтернативный клиент на Python (без UI).
*   Механика **Push-to-Talk** (через библиотеку `keyboard`).
*   Звуковая обратная связь: `AudioCues` генерирует numpy-sweep-тоны при нажатии и отпускании кнопки записи.

## 9. Установка и Запуск

### Требования к системе
*   Docker и Docker Compose
*   NVIDIA GPU + **NVIDIA Container Toolkit**
*   Запущенная база Neo4j

### 9.1 Конфигурация и переменные окружения

Настройки проекта имеют двухуровневую иерархию (Pydantic Settings):
1. **Базовые настройки:** Хранятся в файле `server/config.yaml` (выбор моделей, порты, параметры тайм-аутов, пути к промптам).
2. **Секреты (API ключи, пароли):** Задаются через переменные окружения (в файле `.env`) и переопределяют значения из `.yaml`.

Для задания вложенных параметров из `config.yaml` через `.env`, используется **двойное подчёркивание** (`__`). Например, чтобы задать `api_key` для профиля `gemini` внутри секции `llm -> profiles -> gemini`, переменная должна называться `LLM__PROFILES__GEMINI__API_KEY`.

Скопируйте пример конфига:
```bash
cp .env.example .env
```

**Основные переменные, которые необходимо заполнить в `.env`:**

| Переменная | Назначение |
| :--- | :--- |
| `NEO4J__URI` | Адрес графовой БД (например, `bolt://localhost:7687`) |
| `NEO4J__USER` / `NEO4J__PASSWORD` | Логин и пароль от базы Neo4j |
| `LLM__PROFILES__GEMINI__API_KEY` | Ключ для работы с Gemini API напрямую |
| `LLM__PROFILES__GEMINI__BASE_URL` | Базовый URL для Gemini API |
| `LLM__PROFILES__OTHER__API_KEY` | Ключ от OpenRouter (или другого OpenAI-совместимого провайдера) |
| `LLM__PROFILES__OTHER__BASE_URL` | Базовый URL провайдера моделей (например, `https://openrouter.ai/api/v1`) |
| `TTS__CLOUD__API_KEY` | Ключ для работы с облачным режимом TTS |
| `TTS__CLOUD__BASE_URL` | Базовый URL провайдера облачного TTS |

В файле `server/config.yaml` также присутствует корневой параметр `run_id`. Он используется для привязки и фильтрации данных конкретного запуска эксперимента в графовой БД — все связи, с которыми работает система, маркируются этим идентификатором.

> **Совет:** Чтобы переключить активную модель LLM или TTS, откройте `server/config.yaml` и измените значения `llm.current_profile` (или `tts.mode`). 
> 
> **Обратите внимание:** профиль `other` в конфиге выступает как универсальный клиент для любых **OpenAI-совместимых провайдеров** (например, OpenRouter, vLLM, LM Studio). Вы можете свободно указывать в нём нужную модель (`model`) и гибко менять её настройки: температуру (`temperature`), лимит токенов (`max_output_tokens`) и количество ходов агента (`max_turns`).

### 9.2 Важное замечание по CUDA-драйверам
> [!WARNING]
> По умолчанию проект (`server/Dockerfile`) собирается под **CUDA 12.8.1** (образ `nvidia/cuda:12.8.1-cudnn-runtime-ubuntu22.04`).
> Если ваши драйверы на хост-машине поддерживают только более старую версию CUDA (например, **12.1**), вам **необходимо** внести изменения в `server/Dockerfile` перед сборкой:
> 
> 1. Измените базовый образ в 1 строке.
> 2. Измените index-url пакета PyTorch (например, `cu121`).
> 
> **Пример для CUDA 12.1:**
> ```dockerfile
> FROM nvidia/cuda:12.1.1-cudnn8-runtime-ubuntu22.04
> ...
> RUN pip install --pre torch torchaudio --index-url https://download.pytorch.org/whl/nightly/cu121
> ```

## 10. Тестирование и генерация тестов

В проекте предусмотрен подход к тестированию базы знаний через использование LLM с большим контекстным окном.

1. **Дамп базы данных:**
   Вы можете создать дамп текущего состояния графа (узлы и связи) для конкретного запуска, используя скрипт `tests/dump_db.py`. В качестве аргумента передается `run_id`:
   ```bash
   python -m tests.dump_db <run_id>
   ```
   Скрипт выгрузит данные в JSON-файл вида `db_dump_<run_id>.json`.

2. **Генерация и валидация тестов:**
   Полученный JSON-файл с дампом можно загрузить в LLM с поддержкой большого контекста (от 1 миллиона токенов). На основе этих данных можно попросить модель сгенерировать тестовые сценарии, а также провести валидацию корректности структуры и связей графа прямо в контексте модели.
